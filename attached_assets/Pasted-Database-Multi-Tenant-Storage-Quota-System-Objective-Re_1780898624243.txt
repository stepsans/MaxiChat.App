Database Multi-Tenant & Storage Quota System
Objective

Refactor sistem menjadi arsitektur multi-tenant yang scalable, di mana setiap tenant memiliki database dan alokasi storage masing-masing.

1. Multi-Tenant Database Architecture

Ubah arsitektur database menjadi tenant-based.

Requirements
Setiap tenant memiliki database terpisah (database isolation).
Semua data tenant harus terikat ke tenant yang bersangkutan.
Tidak boleh ada data sharing antar tenant.
Saat tenant dibuat, sistem otomatis membuat database/storage namespace untuk tenant tersebut.
Dashboard, Analytics, Reports, dan seluruh query hanya menampilkan data milik tenant yang sedang login.
Example

Tenant A:

Database: tenant_a

Tenant B:

Database: tenant_b

Tenant C:

Database: tenant_c

Semua data chat, kontak, analytics, campaign, dan logs harus tersimpan pada database tenant masing-masing.
