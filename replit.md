# MaxiChat

Indonesian-language WhatsApp AI automation: a hosted dashboard where each user pairs a WhatsApp account and a Baileys-driven backend answers customer chats via a configurable AI + visual chatbot flow.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 5000)
- `pnpm --filter @workspace/api-server run test` — run api-server unit tests (node:test via tsx; `*.test.ts` files; logic under test must stay db-free)
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- Required env: `DATABASE_URL` — Postgres connection string

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- API: Express 5
- DB: PostgreSQL + Drizzle ORM
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec)
- Build: esbuild (CJS bundle)

## Product

- AI Review requires a per-group "Instruksi AI" (free-text prompt). It is mandatory: without an instruction the module does nothing (the guard in `runReviewForConfig` throws, and create/save is blocked). There is no built-in default receipt-OCR fallback. The output contract is always enforced on top of the instruction: the model must reply with a JSON **array** of objects keyed by the exact column names — **one object per receipt line item**, so a single nota with N items produces N Sheet rows (whole-nota fields are repeated on each row). Whole-receipt notas (no line items) yield a one-element array. A one-click "Laporan Kas Harian" template fills the instruction + matching per-item columns. `lastRunCount` / the status pill now count Sheet rows ("baris ditulis"), not notas. A "Generate by AI" button (in Output settings, below the spreadsheet selector and above the columns editor) calls `POST /ai-review/generate-columns`, which reads the Instruksi AI and asks the model to propose the output columns ({name, hint}); it's super-admin-only and records AI usage against the owner. It replaces the current columns; the instruction is still mandatory.
- AI Review dedup is purely time-watermark based (`lastRunAt`): each receipt is read once by arrival time (`createdAt > lastRunAt && <= now`); the watermark only advances on a successful run. Items are processed **regardless of direction** (inbound AND outbound) — in a "laporan kas" group the receipts are frequently posted by the paired number itself (owner forwarding nota), which is recorded as `outbound` (fromMe). Filtering to inbound-only silently dropped every owner-posted receipt. The bot never sends images into these groups, so including outbound only adds genuine human-posted photos.
- AI Review accepts both **photos and document attachments**: `mediaType="image"`, plus `mediaType="document"` whose mime is `application/pdf` (PDF invoices) or `image/*` (a photo sent "as file"). Other document kinds (xlsx/docx/zip/audio) are excluded. PDFs are sent to the model as a chat.completions `file` content part (the model reads the PDF directly — no local rasterization); photos use `image_url`. PDF input is **provider-gated**: only OpenAI models (the Replit-managed default + BYOK OpenAI) — Gemini/OpenRouter PDFs are skipped (errors++ + warn). PDFs over 15 MB are skipped. On Drive upload, PDFs skip the image-scanner (`scanDocument`) and are archived as-is.
- AI Review number contract: the global output contract spells out Indonesian numeric convention (`.`=thousands, `,`=decimal) with examples — so "34.000" extracts as `34000`, not `34`. The per-group instruction does NOT need to repeat this. Without it, models misread the thousands dot as a decimal and write `34` to the Sheet. The Sheet is append-only and never read back — **deleting a row in Google Sheets does NOT cause re-extraction**, and already-appended notas are never re-read. Pure parsing helpers (`parseJsonRows`/`toRowObjects`/`cellToString`) live in db-free `ai-review-parse.ts` so they're unit-testable.
- AI token usage is tracked per tenant owner (super admin). Each owner uses their own AI quota; usage by team members rolls up to the owner. The monthly reporting period is anchored on the owner's join date (day-of-month), not the 1st. Visible in the admin app (all owners) and the whatsapp-ai dashboard (each owner's own). No historical backfill — usage accrues forward only.
- **Outbound blue ticks**: MaxiChat mirrors WhatsApp's delivery/read ticks on messages WE sent. `chat_messages.status` (text: `sent`|`delivered`|`read`; null = not yet observed, treated as sent) is advanced **forward-only** by the SAME two Baileys listeners as the inbound read sync, but on `key.fromMe === true` events (the reverse direction). db-free parsers in `lib/chat-read-sync.ts`: `outboundStatusFromMessageUpdate` (DELIVERY_ACK(3)→delivered, READ(4)/PLAYED(5)→read) + `outboundStatusFromReceiptUpdate` (read/playedTimestamp→read, receiptTimestamp-only→delivered). `applyOutboundStatusSignal` in `routes/whatsapp.ts` updates by (chat's channel+phone, wa_message_id) with a SQL rank guard (sent<delivered<read) so an out-of-order `delivered` can never undo a `read`. The chat bubble (`whatsapp-ai/ConversationPane.tsx`) renders a single grey check (sent/null), double grey check (delivered), or double blue check (read). Status column applied via raw `psql`; unit-tested in `lib/chat-read-sync.test.ts`. This is the OUTBOUND counterpart deliberately left out of the inbound cross-device read sync below.
- **Cross-device read sync**: when a customer chat is read on ANY linked WhatsApp device, MaxiChat clears its own unread badge within ~5s (existing chat-list poll — propagation is fully poll-backed via the 5s `useListChats` refetch, so this is backend-only). All read paths (chat-meta `unreadCount:0`, live read receipts, message-status updates) converge on ONE causally-guarded clear `clearUnreadUpTo(channelId, phoneNumber, readUpTo)` in `routes/whatsapp.ts`: clears only when the read point covers the chat's latest message (`lastMessageAt IS NULL OR lastMessageAt <= readUpTo`), so a stale/out-of-order read can never wipe a NEWER unread and positive counts are never mirrored. Two Baileys event sources (db-free parsers in `lib/chat-read-sync.ts`, unit-tested): `message-receipt.update` carries an explicit `readTimestamp`/`playedTimestamp` → used as `readUpTo`; a timestamp-less receipt is treated as a NON-read (delivery/etc.) and returns null (must NOT clear). `messages.update` raising an inbound msg to READ(4)/PLAYED(5) carries no timestamp → message-anchored on the referenced msg's `createdAt`. Both **skip `key.fromMe`** (outbound blue-ticks = customer read what we sent = out of scope). Listeners are epoch-guarded with quiet `logger.debug` observational logging. Out of scope: outbound blue-tick mirroring, Telegram, websocket/push, reverse-direction phone marking.
- Customer labels are **contact-level**, not per-chat: a label is keyed by (owner, phone number) in `contact_labels`, so labeling a number "High Risk" in one channel makes it show on the same number in every other channel of that owner — and on chats created later for that number. The Chat list shows each chat's label chips + a label filter; Dashboard and Analytics show chat counts per label (a contact present in 2 channels counts as 2 chats). Telegram chats use a `tg:<id>` phone key so they never cross-link with WhatsApp numbers.
- **Reset Database Tenant** (Settings → Database, super-admin only): wipes ALL of one tenant's *operational* data so they can "start fresh" — chats + messages, contact/customer labels, analytics snapshots (`usage_snapshots`), AI usage logs (`ai_usage_events`), and every uploaded file (`media_objects` ledger rows + the Object Storage blobs under `tenants/<owner>/`). It deliberately does NOT touch the account, subscription, plan, quota, channels, settings, or products, and never crosses tenants. AI "memory" is derived live from chat history (no separate table), so wiping chats clears it implicitly. Endpoints: `POST /database/reset` (`requireSuperAdmin`, owner = `resolveOwnerUserId`) + `GET /database/reset-audit` (`routes/database.ts`); core logic in `lib/tenant-reset.ts`. **Ordering is load-bearing** (see `lib/tenant-reset.ts` header): Object Storage is swept FIRST via `deleteTenantPrefix(owner)` (one prefix sweep = exact `files` count; best-effort, retention reconciles any leftover blobs), THEN every DB delete + the `tenant_reset_audit` insert run in ONE `db.transaction` so DB state and the audit row are all-or-nothing. Blobs-first means a DB rollback leaves recoverable orphan *ledger rows*, not orphan *blobs*. UI (`whatsapp-ai/src/pages/Settings.tsx` `DangerZoneCard`) requires a TWO-step confirmation (type `RESET` → final "are you sure" dialog) and shows the last-5 audit history. `tenant_reset_audit` table applied via raw `psql` (no drizzle push). DB-backed regression test: `lib/tenant-reset.test.ts` (mocks ObjectStorageService; asserts owner-only wipe + foreign tenant intact + audit row).

## Subscription (Hybrid model — in progress)

- Self-serve **Hybrid** subscription: prepaid base **plan** + paid **add-ons/top-ups** (extra users, channels, AI tokens). Gateway = **Xendit** (VA/QRIS/e-wallet). Add-ons may exceed plan quotas.
- Plans & add-ons are **admin-configurable DB catalog rows**, never hardcoded: `plans` (`key` UNIQUE = `users.plan`, name, priceIdr, durationDays, quotaUsers/Channels/Tokens, isActive, sortOrder) + `addons` (type token/channel/user_seat, unitAmount, priceIdr). `tenant_quota` holds per-owner **limits only** (plafon = plan quota + add-on top-ups); actual usage is computed live (reuses the metered billing engine). `payments` is the purchase ledger (`external_id` = Xendit ref for idempotent webhooks). All money is whole-integer Rupiah.
- Per-plan invited-seat cap (`agents.ts`) reads `plans.quotaUsers` by the owner's raw `plan` key; the old hardcoded `basic/pro/business/enterprise` map is a fallback only.
- Migrations applied via raw `psql` (repo keeps no drizzle migration history); never `drizzle-kit push`.
- Admin catalog CRUD (FASE 1): platform-admin-only endpoints `/admin/plans` + `/admin/addons` (under `requireAdmin`, distinct from tenant super_admin) managed from the admin app's "Paket & Add-on" tab. Generated zod can't fully enforce the contract, so `routes/plans.ts` adds two server guards: (1) integer re-check (OpenAPI `integer` codegens to `zod.number()`, accepts decimals); (2) plan delete is blocked with 409 when any `users.plan` references its `key` (archive via `isActive=false` instead) — add-ons delete freely. Duplicate plan key → 409 both via pre-check and Postgres `23505` catch (race-safe). Plan `key` is immutable (omitted from update body).
- Xendit credentials are **admin-configurable** from the admin app's "Gateway Pembayaran" tab (platform-admin only, under `requireAdmin`). `payment_gateway_config` is a singleton DB row (UNIQUE on `provider`, currently only `xendit`) holding the operator's ONE gateway account (NOT per-tenant) with AES-256-GCM-encrypted `secret_key_enc`/`callback_token_enc` (reuses `lib/crypto.ts`, key from `SESSION_SECRET`). `lib/payment-config.ts` resolves credentials **DB-first with per-field env fallback** (`XENDIT_SECRET_KEY`/`XENDIT_CALLBACK_TOKEN`); a row with `is_active=false` is treated as if absent (env fallback only). The Xendit getters in `lib/xendit.ts` are now **async** (read through the store). Reads are **masked** — the API never returns raw secrets, only `secretKeyLast4` + configured flags + source (db/env). Endpoints `GET/PUT /admin/payment-config` (`routes/payment-config.ts`). `updatePaymentConfig` is an **atomic INSERT…ON CONFLICT(provider) DO UPDATE** that only sets changed columns (omitted creds preserved; `clear*` flags null them) so concurrent first writes can't race the unique index. The webhook URL shown in the UI is `${origin}/api/webhooks/xendit`. Table applied via raw `psql` (no drizzle push).
- Xendit payments (FASE 2, backend only — tenant UI is FASE 5): tenant endpoints `GET /billing/catalog` (active plans+addons), `GET /billing/quota` (limits + live usage), `POST /billing/checkout` (server computes the amount from the catalog, inserts a `pending` payment, creates a hosted Xendit invoice, returns its URL), `GET /billing/payments` (history). `enforceSubscription` exempts `/billing` so EXPIRED tenants can still renew. `lib/xendit.ts` is a thin Invoice API client (secret key as Basic-auth username, fixed API base = no SSRF) + constant-time `verifyXenditCallbackToken`. `lib/subscription-purchase.ts` holds the purchase-application logic (activate plan, add-on top-up, dispatch by kind). Inbound webhook `POST /webhooks/xendit` is mounted **before** `requireAuth` (authed by the `x-callback-token` header / `XENDIT_CALLBACK_TOKEN`); it reconciles **before** ACKing and 500s on failure so Xendit retries. **Settlement is atomic**: the `pending→paid` flip + entitlement grant run in one `db.transaction` (conditional `WHERE status='pending'` = idempotency guard) — a failed grant rolls back to `pending` (retriable) so a tenant is never charged without quota. Requires secrets `XENDIT_SECRET_KEY` + `XENDIT_CALLBACK_TOKEN`. See `artifacts/api-server/docs/payments-webhook.md`. The webhook reconciliation lookups (both the invoice-id match and the `maxichat-pay-<id>` fallback) are scoped to `provider='xendit'` — manual payments also carry a `maxichat-pay-<id>` externalId (the customer's code), so an unscoped fallback would let a valid Xendit callback settle a manual order the operator never confirmed. The poller is the ONLY path allowed to settle manual rows.
- Cart checkout + PDF invoice (FASE 6): the tenant Billing checkout is now a **cart**, not one-click buy. Buttons say "Tambah ke Keranjang" (`CheckoutSection.tsx`): plans are **single-select** (adding a plan replaces any plan already in the cart; qty forced to 1), add-ons add with qty N. The cart is **one order = one `payments` row** with `kind="cart"` + a `line_items jsonb` snapshot (`PaymentLineItem[]` = `{kind,refId,quantity,name,unitPriceIdr,lineAmountIdr}`, mirrored in `schema/payments.ts` with `.$type<>()`). `POST /billing/checkout` now takes `{items:[CheckoutItem{kind,refId,quantity}], successRedirectUrl}`, validates (≥1 item, ≤1 plan, plan qty=1, `Number.isInteger` refId+qty), snapshots catalog prices server-side, computes the total, and inserts ONE pending payment for both manual + Xendit paths (one invoice / one Sheet row). Settlement: `applyPaidPayment` has a `"cart"` branch that **applies plans BEFORE add-ons** (plan activation RESETS quota to the plan base, so an add-on top-up applied first would be wiped) — all inside the existing atomic `settlePaymentPaid` transaction. Legacy `plan/addon/renewal` settlement branches stay for old rows; `lineItems` is nullable on `PaymentRecord`. **PDF invoice**: `GET /billing/payments/:id/invoice` (owner-scoped, 404 non-enumerable otherwise) returns a binary `application/pdf` built with **pdf-lib** (`lib/invoice-pdf.ts`, A4 line-items table + total + manual bank box; reuses `quotation-pdf.ts` patterns; synthesizes a single line for legacy empty-`lineItems` rows). This endpoint is **deliberately NOT in OpenAPI** (binary response); the frontend downloads via raw `fetch('/api/billing/payments/:id/invoice',{credentials:'include'})`→blob, matching the avatar-upload precedent. Migration (`line_items` column) applied via raw `psql` (no drizzle push).
- Manual payment gateway + tenant checkout UI (FASE 5): the operator picks the active provider (`xendit` | `manual`) in the admin "Gateway Pembayaran" tab. `payment_method_settings` is a **singleton operator-level** table (id=1 CHECK; NOT per-tenant) holding the active provider + manual bank fields (Nama Bank/No Rekening/Nama Pemilik — NOT secret, returned to customers) + a verification Google Sheet config (credentialId FK ON DELETE SET NULL / spreadsheetId / tab). Xendit creds stay in `payment_gateway_config`. Helpers in `lib/manual-payment-config.ts`; `lib/manual-payment-sheet.ts` does ensureHeader/appendOrderRow/readAndSettleManualPayments (reuses the /credentials OAuth client); `lib/manual-payment-poller.ts` is a 60s scheduler (inFlight guard, `lastPolledAt` liveness write) wired in index.ts. Manual model is **"Otomatis"**: on manual checkout the system writes a PENDING order row to the Sheet (header `Kode Pembayaran | Tanggal | Nama Tenant | Email | Item | Jumlah (Rp) | Status | Catatan`); the operator flips the **Status** cell to `LUNAS`; the poller detects it and settles via the same idempotent `settlePaymentPaid`. Manual reuses the `payments` table (provider='manual', externalId=`maxichat-pay-<id>` = the "Kode Pembayaran"; no payments schema change). Manual checkout **fails fast**: it requires BOTH bank details (`isManualBankConfigured`) AND a configured Sheet (`isVerificationConfigured`), and if the Sheet row append throws it marks the pending row `failed` + returns 502 (no orphan pending / no "paid into the void"). Checkout enforces whole-integer Rupiah at the boundary (OpenAPI `integer` codegens to `zod.number()` which accepts decimals, so the route re-checks `Number.isInteger` on `refId` + `quantity`). `CheckoutResult` is a `mode`-discriminated union: `xendit`→`invoiceUrl` to redirect; `manual`→`code`+bank fields for the transfer-instructions dialog. Tenant UI: `whatsapp-ai/src/components/CheckoutSection.tsx` (plans+addons buy buttons → `POST /billing/checkout`, branch on `result.mode`, payment-history table) embedded in `pages/Billing.tsx`; reads active method via `GET /billing/payment-method`. Migration applied via raw `psql` (no drizzle push).
